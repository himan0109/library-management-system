1 · Introduction & Overview

The Library Management System is a full-stack Spring Boot web application that manages two core entities — Authors and Books — and exposes Create, Read, and Update operations through a browser-based JSP interface.

RequirementImplementation
Two entities with a relationshipAuthor (1) ↔ Book (Many) via @OneToMany / @ManyToOne
Populate databaseDataInitializer seeds 10 authors + 10 books on startup
Create operationForm-based POST with Bean Validation + integrity error handling
Read operationList views + custom INNER JOIN JPQL query returning a DTO
Update operationPre-filled edit form, POST updates, redirect-after-post
Unit tests32 tests across AuthorRepository, BookRepository, AuthorService, BookService

2 · Entity Relationship Design

The two entities share a One-to-Many relationship: one Author can write many Books, and each Book belongs to exactly one Author.

AUTHORS
PK id BIGINT AUTO
name VARCHAR(100) NOT NULL
email VARCHAR(150) UNIQUE NOT NULL
nationality VARCHAR(60) NOT NULL
birth_year INT NOT NULL
1
⟷
Many author_id FK
BOOKS
PK id BIGINT AUTO
title VARCHAR(200) NOT NULL
isbn VARCHAR(20) UNIQUE NOT NULL
published_year INT NOT NULL
genre VARCHAR(80) NOT NULL
FK author_id BIGINT NOT NULL
Cardinality: One Author → Many Books. The author_id foreign key in the BOOKS table enforces referential integrity at the database level. Cascading ALL on the Author side ensures books are removed when their author is deleted.

4 · Implementation — Entity Layer

Author.java

@Entity
@Table(name = "authors")
public class Author {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @NotBlank @Size(max = 100)
    @Column(nullable = false, length = 100)
    private String name;

    @Email @NotBlank
    @Column(nullable = false, unique = true, length = 150)
    private String email;

    @NotBlank
    private String nationality;

    @Min(1000) @Max(2025)
    private int birthYear;

    @OneToMany(mappedBy = "author", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Book> books = new ArrayList<>();
}

Book.java

@Entity
@Table(name = "books")
public class Book {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @NotBlank @Size(max = 200) private String title;

    @NotBlank @Pattern(regexp = "^[0-9\\-]{10,17}$")
    @Column(unique = true) private String isbn;

    @Min(1000) @Max(2025) private int publishedYear;

    @NotBlank private String genre;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "author_id", nullable = false)
    private Author author;
}

Key JPA annotations used:

@Column(unique=true)Adds a UNIQUE constraint (email, ISBN)
@OneToManyAuthor owns the one side of the relationship
@ManyToOne @JoinColumnBook holds the FK column author_id
@NotBlank @Email @Min @MaxBean Validation constraints checked before persistence

5 · Implementation — Repository Layer

Both repositories extend JpaRepository<T, ID>, which automatically provides save(), findById(), findAll(), deleteById(), and more.

Custom query methods

// AuthorRepository
Optional<Author> findByEmail(String email);
List<Author> findByNationality(String nationality);
boolean existsByEmail(String email);

@Query("SELECT a FROM Author a WHERE LOWER(a.name) LIKE LOWER(CONCAT('%',:name,'%'))")
List<Author> searchByName(String name);

Custom INNER JOIN query (BookRepository)

@Query("""
    SELECT new com.bda.library.dto.BookWithAuthorDTO(
        b.id, b.title, b.isbn, b.publishedYear, b.genre,
        a.id, a.name, a.nationality)
    FROM Book b INNER JOIN b.author a
    ORDER BY a.name, b.title
    """)
List<BookWithAuthorDTO> findAllBooksWithAuthorDetails();

// Fetch-join avoids N+1 problem when loading the books list
@Query("SELECT b FROM Book b INNER JOIN FETCH b.author a ORDER BY b.title")
List<Book> findAllWithAuthor();
The JPQL INNER JOIN query projects results into a BookWithAuthorDTO object (using the new constructor expression), satisfying the requirement for a custom join query between both entities. The SQL equivalent is: SELECT b.*, a.name, a.nationality FROM books b INNER JOIN authors a ON b.author_id = a.id

6 · Implementation — Service Layer

Service classes implement their respective interfaces and are annotated with @Service @Transactional. Read-only operations use @Transactional(readOnly = true) for performance.

AuthorServiceImpl — update method

@Override
public Author update(Long id, Author updated) {
    Author existing = authorRepository.findById(id)
        .orElseThrow(() -> new EntityNotFoundException("Author not found: " + id));
    existing.setName(updated.getName());
    existing.setEmail(updated.getEmail());
    existing.setNationality(updated.getNationality());
    existing.setBirthYear(updated.getBirthYear());
    return authorRepository.save(existing);
}

BookServiceImpl — save method (author resolution)

@Override
public Book save(Book book) {
    // Re-fetch the full Author entity by the ID submitted from the form
    Author author = authorRepository.findById(book.getAuthor().getId())
        .orElseThrow(() -> new EntityNotFoundException("Author not found"));
    book.setAuthor(author);
    return bookRepository.save(book);
}

Create handler with validation & integrity error handling

@PostMapping
public String createAuthor(@Valid @ModelAttribute("author") Author author,
                           BindingResult bindingResult, Model model,
                           RedirectAttributes redirectAttrs) {
    if (bindingResult.hasErrors()) {          // Bean Validation failed
        model.addAttribute("formAction", "/authors");
        return "authors/form";               // redisplay with error messages
    }
    try {
        authorService.save(author);
        redirectAttrs.addFlashAttribute("successMessage", "Author created.");
    } catch (DataIntegrityViolationException e) {
        model.addAttribute("errorMessage", "Email already exists.");
        return "authors/form";
    }
    return "redirect:/authors";             // Post-Redirect-Get pattern
}

The same pattern is applied for Books, with duplicate ISBN detected via DataIntegrityViolationException.

10 · Exception Handling

A @ControllerAdvice class — GlobalExceptionHandler — catches exceptions thrown anywhere in the controller layer and renders a friendly error page.

ExceptionCauseUser message
DataIntegrityViolationExceptionDuplicate email or ISBN in the databaseInline error on the form (not redirected to error page)
EntityNotFoundExceptionInvalid ID in URL (e.g. /authors/999/edit)Error page: "Record Not Found"
Exception (generic)Any unexpected runtime errorError page: "Unexpected Error"
@ControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(DataIntegrityViolationException.class)
    public String handleDataIntegrity(DataIntegrityViolationException ex, Model model) {
        model.addAttribute("errorMessage", "Constraint violation — check for duplicate values.");
        return "error";
    }
}

@Test

All 32 tests use AssertJ fluent assertions for readable failure messages. Repository tests use the @DataJpaTest slice which boots only the JPA layer against a real H2 database — ensuring queries are validated against an actual SQL engine.

13 · Challenges Faced & Solutions

ChallengeSolution
N+1 Query Problem — loading books list triggered a separate SQL query per book to fetch the author. Added a INNER JOIN FETCH JPQL query (findAllWithAuthor()) in BookRepository so the author is loaded eagerly in one SQL statement.
Author binding in Book form — the form submits only author.id, but JPA needs a managed Author entity. The BookServiceImpl.save() method re-fetches the full Author entity from the repository using the submitted ID before calling bookRepository.save().
Duplicate constraint errors — H2 throws a low-level DataIntegrityViolationException when a duplicate email or ISBN is inserted, without a user-friendly message. Caught in the controller's try-catch block and in GlobalExceptionHandler; a human-readable message is added to the model and the form is re-rendered.
Reusing the same JSP form for Create and Update — the form action URL differs between the two operations. Controllers pass a formAction string attribute to the model. The JSP's <form:form action="${formAction}"> uses this dynamically, so one template handles both cases.
JPQL DTO projection — mapping a multi-table query result to a plain Java object without involving a full entity. Used the JPQL new com.bda.library.dto.BookWithAuthorDTO(...) constructor expression, passing selected fields directly into the DTO constructor.
